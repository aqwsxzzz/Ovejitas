## Architecture / Design Changes

### Inline edit/delete pattern in `IndividualList`

`individual-list.tsx` was rewritten to support an inline edit/delete pattern rather than navigating to a separate detail route for mutations:

- `editingId: string | null` local state tracks which row is in edit mode.
- When `editingId` matches a row's ID, an `IndividualForm` is rendered inline inside the row.
- Submitting the form calls `onUpdateIndividual(individual, data): Promise<void>` prop, which is wired to `useUpdateIndividual` in `flock-detail-page.tsx`.
- Delete replaced with an inline two-step confirmation block (no Dialog, no `window.confirm`).
- Prop interface change: `onEditIndividual` removed; `onUpdateIndividual` and `isUpdating` added.

### Cache invalidation broadened

`livestock-queries.ts` was updated to use stable prefix-based invalidation keys rather than exact filter-keyed entries:

```
// Before (too specific — missed filtered variants)
queryClient.invalidateQueries({ queryKey: [...livestockQueryKeys.all, "individualsByAsset", farmId, assetId, filters] })

// After (hits all variants)
queryClient.invalidateQueries({ queryKey: [...livestockQueryKeys.all, "individualsByAsset", farmId, assetId] })
```

`useUpdateIndividual` also performs a `setQueryData` on the cached list for immediate optimistic-style feedback before the invalidation completes.

## API and Data Contract Changes

- **`birth_date` excluded from individual update payload**: the backend `UpdateIndividualPayload` type does not include `birth_date`. The field was removed from the update call in `flock-detail-page.tsx`. It remains displayed in read-only form but cannot be edited via inline form.
- No other API contract changes.

## Implementation Notes

### `flock-detail-page.tsx`
- Imports `useUpdateIndividual` and wires `handleUpdateIndividual` callback.
- `handleUpdateIndividual(individual, data)` calls `updateIndividualMutation.mutateAsync` with `name / tag / status / mother_id / father_id / extra`.
- Passes `onUpdateIndividual={handleUpdateIndividual}` and `isUpdating={updateIndividualMutation.isPending}` to `<IndividualList>`.
- Removed `edit` search param from non-edit navigate calls to avoid accidental edit-mode activation.

### `individual-detail-page.tsx`
- `isMutating` now combines `updateMutation.isPending || deleteMutation.isPending`.
- `startInEditMode` prop sourced from validated `edit` boolean search param.

### `v2-log-page.tsx`
- Added `resolveReturnPath` helper: if the stored return path is `/v2/log` or empty, falls back to `/v2`.
- All `navigate` calls after completing a quick action use `replace: true`.

### `v2.production-units.flock.$unitId.individuals.$individualId.tsx`
- Added `edit: z.boolean().optional()` to search params schema.
- Passes `startInEditMode={search.edit ?? false}` to `IndividualDetailPage`.

### i18n
- All user-facing English strings replaced with Spanish across:
  - `individual-form.tsx` — field labels, placeholders, select options, error messages
  - `individual-detail.tsx` — section headers, button labels, delete confirmation
  - `genealogy-tree.tsx` — relation labels, sex values, inbreeding warning, empty state
  - `individual-detail-page.tsx` — back button, error message
- `livestock-page.tsx` — Ganado card: description hidden from list view; `MapPin` icon added before location text.

## Code Examples

### Inline edit row in `IndividualList`

```tsx
{editingId === individual.id ? (
  <IndividualForm
    individual={individual}
    farmId={farmId}
    assetId={assetId}
    onSubmit={(data) => onUpdateIndividual?.(individual, data)}
    onCancel={() => setEditingId(null)}
    isSubmitting={isUpdating}
  />
) : (
  // read-only row with Edit / Delete buttons
)}
```

## Risks and Mitigations

| Risk | Mitigation |
|---|---|

| `individual-list.tsx` rewrite could regress list render | Build passes, manual verification of list rendering recommended |
| Cache prefix broadening could over-invalidate on unrelated mutations | Acceptable — stale-while-revalidate pattern handles gracefully |
