## Architecture / Design Changes

### Public landing entrypoint

The app now exposes a dedicated public landing page on both `/` and `/v2`. This keeps
the V2 entry experience separate from the authenticated shell while preserving the
existing login/signup routes.

### Auth guard behavior

The root route guard was updated to treat landing paths as public for unauthenticated
visitors. Authenticated users still resolve to the dashboard flow, so the landing page
remains an entry point rather than a dead-end.

### Shell exclusion for landing paths

`V2ShellLayout` now opts out for `/v2` and `/v2/` so the marketing-style landing page
does not render the bottom navigation or private app shell chrome.

## Implementation Notes

### Landing page composition

`v2-landing-page.tsx` mirrors the supplied handoff with localized Spanish copy,
CTA-driven routing to login/signup, a product preview section, and feature/value cards.

### Asset handling

The preview image used in the mock was downloaded into `public/landing/` so the page
ships without relying on an external runtime image host.

### Background treatment

The landing page background is implemented in code with layered gradients and a small
noise texture data URI to approximate the handoff without external dependencies.

## Risks and Mitigations

| Risk | Mitigation |
|---|---|
| The landing page may drift from the handoff if assets are replaced later | Keep visual tweaks contained to the landing page module and public asset folder |
| Local static asset size increases the bundle footprint slightly | The image is served from `public/` and avoids runtime network dependency |
| Route guard changes could affect authenticated entry behavior | Guard logic preserves the existing dashboard redirect for signed-in users |
