## Architecture / Design Changes

### Shared V2 auth page shell

Auth entry screens now share a dedicated `V2AuthPageFrame` wrapper instead of duplicating page structure in each route. This centralizes layout, visual hierarchy, and action placement for login/signup while keeping the form components focused on form concerns.

### Livestock event flow refresh

Livestock event detail interactions were tightened so unit-level event mutations and follow-up reads stay aligned with the current asset context. The changes are centered in query wiring and page-level orchestration rather than introducing a new state model.

### Event-type-aware report filters

Reports filtering now treats `unit` and `bucket` as conditional inputs rather than universal filters:

- `unit` is only sent for production reports.
- `bucket` is hidden for individual scope.
- UI visibility and outbound filter payloads now follow the same rules, reducing invalid query combinations.

---

## API and Data Contract Changes

- **DELETE response tolerance**: `axios-helper.ts` now accepts empty delete responses without treating them as malformed payloads. This is a frontend transport hardening change, not a backend contract expansion.
- **Report request params narrowed**: production report queries no longer send `unit` for non-production event types.
- No new endpoints were added in this release.

## Implementation Notes

### Auth
- `v2-login-page.tsx` and `v2-signup-page.tsx` were reworked to use the shared auth frame.
- `login-form.tsx` and `sign-up-form.tsx` were updated to match the V2 page layout and localized copy requirements.
- `src/routes/_public/_layout.tsx` was adjusted so the public auth experience uses the updated shell consistently.

### Livestock
- `livestock-queries.ts` was updated to support the corrected asset-event detail refresh path.
- `unit-event-form.tsx` and `flock-detail-page.tsx` now align submission and refresh behavior with the active livestock context.

### Reports
- `reports-filter-panel.tsx` now computes `shouldShowBucketFilter` and `shouldShowUnitFilter` from scope and event type.
- `production-report.tsx` derives a `unitFilter` so only valid report requests include the parameter.
- `v2.reports.tsx` applies a fallback unit for the cost-per-unit report when no report-level unit filter is active.

## Code Examples

### Conditional unit forwarding for reports

```tsx
const unitFilter = eventType === "production" ? unit : undefined;

useProductionReport({
  farm_id: farmId,
  event_type: eventType,
  unit: unitFilter,
});
```

### Conditional filter payload from the report panel

```tsx
onApply({
  scope,
  eventType,
  bucket,
  unit: shouldShowUnitFilter ? unit : undefined,
  dateFrom: dateFromApi,
  dateTo: dateToApi,
});
```

## Risks and Mitigations

| Risk | Mitigation |
|---|---|
| Auth page redesign could shift layout behavior on small screens | Shared frame keeps layout logic centralized; build completed and manual smoke is recommended on mobile widths |
| Livestock event refresh changes could miss a stale data edge case | Changes were limited to the existing query/mutation path rather than adding new caches |
| Reports filter narrowing could alter previously tolerated invalid requests | UI visibility and API payload generation were updated together to keep behavior consistent |
